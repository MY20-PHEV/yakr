End-to-End Lifecycle

From strangers to delivered messages: invite, pairing, profile exchange, send, and fetch — the complete Yakr session story.

Master Flow

flowchart TB
  Start([Two users want to chat]) --> Invite[1. Alice creates invite URL]
  Invite --> Accept[2. Bob accepts invite]
  Accept --> Safety[3. Verify safety code OOB]
  Safety --> Pair[4. X25519 pairing handshake]
  Pair --> Master[5. Derive master secret + ratchet]
  Master --> Profile[6. Exchange delivery profiles]
  Profile --> Send[7. Alice sends encrypted message]
  Send --> Store[8. Relay stores opaque blob]
  Store --> Fetch[9. Bob fetches and decrypts]
  Fetch --> Done([Conversation established])

Step 1–3: Invite & Verification

Alice generates a signed CBOR invite bundle and shares it as a URL or QR code. Bob decodes it, verifies the Ed25519 signature, and compares the safety code with Alice in person or over a trusted channel.

sequenceDiagram
  participant Alice
  participant Rendezvous as Rendezvous (optional)
  participant Bob
  Alice->>Alice: create_invite() + sign CBOR
  Alice->>Bob: yakr://invite/... (QR, link, NFC)
  Bob->>Bob: verify Ed25519 signature
  Bob->>Bob: compute safety code
  Note over Alice,Bob: Both display 4510 2126 16
  Alice->>Bob: confirm codes match (voice/SMS)
CLI — AlicePhase 4
yakr invite create --port 8090
# → yakr://invite/qWhwcm90b2NvbG... (truncated)
# → Safety code: 4510 2126 16

Group relay rendezvous

When rendezvous_hint points at a group relay, both sides poll outbound — no inbound HTTP on phones. Alice registers on the relay; Bob posts a pairing request; Alice responds. See docs/spec/relay-rendezvous.md.

CLI — relay rendezvousimplemented
# Alice (paired with relay operator beforehand)
yakr invite create --rendezvous https://relay.example:8090 --no-wait
yakr invite relay wait

# Bob (not paired with relay operator)
yakr invite accept "yakr://invite/..." --name alice
CLI — BobPhase 4
yakr invite accept "yakr://invite/qWhwcm90b2NvbG..."
# → Safety code: 4510 2126 16  ✓ match

Step 4–5: Pairing & Session Keys

Bob sends a pairing request with ephemeral X25519 keys (and ML-KEM ciphertext for hybrid invites). Both sides combine shared secrets into a master key and initialise a double-ratchet for forward secrecy on subsequent messages.

sequenceDiagram
  participant Bob as Bob (joiner)
  participant RV as Rendezvous
  participant Alice as Alice (inviter)
  Bob->>RV: POST pairing request
  Alice->>RV: poll / fetch request
  Alice->>RV: POST pairing response
  Bob->>RV: fetch response
  Bob->>Bob: derive master_secret
  Alice->>Alice: derive master_secret
  Note over Alice,Bob: Same master → Contact created

Classical master derivation

HKDFyakr-v0.4
shared   = X25519(alice_agreement, bob_ephemeral)
master   = HKDF-SHA256(
             ikm  = shared,
             salt = SHA256(pairing_transcript),
             info = "yakr/v0.4/pair-master"
           )

Hybrid PQ pairing (optional)

flowchart LR
  X1[X25519 identity shared] --> XS[x_secret]
  X2[X25519 ephemeral shared] --> XS
  PQ[ML-KEM-768 secret] --> HKDF
  XS --> HKDF[HKDF hybrid-master]
  TH[transcript hash] --> HKDF
  HKDF --> MS[master_secret]

Step 6: Delivery Profile Exchange

Each contact publishes a signed profile describing relay URLs, wrap secrets, direct P2P hints, mailbox epoch length, and receipt policy. When Bob changes relays, Alice automatically uses the updated profile on next send.

sequenceDiagram
  participant Bob
  participant Alice
  Bob->>Bob: profile publish (local)
  Bob->>Alice: inner message type=profile
  Alice->>Alice: verify Ed25519 sig, store profile
  Note over Alice: Next send uses Bob's relay descriptors
Route selection: Try direct P2P hints first (2 s timeout), then relay descriptors from the contact's signed profile. Stale profiles trigger refresh and retry.

Step 7–8: Send & Store

Alice builds an inner JSON message, encrypts it with the per-sequence message key, wraps it in an outer blob with a mailbox tag, and posts to the relay (directly or via two-hop onion).

flowchart TB
  IM[Inner message JSON
seq=1 body=hello] --> MK[message_key seq=1]
  MK --> ENC[XChaCha20-Poly1305]
  ENC --> OB[Outer blob]
  MS[master_secret] --> MT[mailbox_secret
direction alice→bob]
  MT --> TAG[HMAC tag epoch=1000]
  TAG --> OB
  OB --> POST[POST /v1/blobs]
  POST --> RELAY[(Mailbox Relay)]

Step 9: Fetch & Decrypt

Bob derives mailbox tags for the current epoch and several lookback epochs, fetches all matching blobs, decrypts with his copy of the master secret, and advances the receive ratchet.

sequenceDiagram
  participant Bob
  participant Relay as Mailbox Relay
  Bob->>Bob: derive tags for epochs N, N-1, N-2
  loop each tag
    Bob->>Relay: GET /v1/blobs/{tag}
    Relay-->>Bob: [{ciphertext, expires_at, ...}]
  end
  Bob->>Bob: decrypt with message_key(seq)
  Bob->>Bob: display "hello vector"
CLI — offline delivery demoPhase 1
# Alice sends while Bob is offline
docker compose run --rm --no-deps alice send bob "hello"

# Bob fetches later
docker compose run --rm --no-deps bob fetch alice

Delivery Receipts (Phase 2+)

After Bob successfully decrypts a message, he may send an encrypted receipt back over the reversed two-hop route so Alice knows delivery succeeded.

sequenceDiagram
  participant Alice
  participant Entry
  participant Mailbox
  participant Bob
  Note over Alice,Bob: Forward path: Alice → Entry → Mailbox → Bob
  Bob->>Bob: create inner message type=receipt
  Bob->>Mailbox: receipt over reversed route
  Mailbox->>Entry: forward
  Entry->>Alice: deliver receipt
