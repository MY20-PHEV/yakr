Delivery & Relays

How messages travel through the social relay network: single-hop, two-hop onion, path rotation, privacy modes, and relay abuse limits.

Phase 1: Single-Hop Delivery

The simplest path. Alice posts directly to Bob's mailbox relay. The relay sees the mailbox tag and ciphertext size, but not who sent the message or what it says.

sequenceDiagram
  participant Alice
  participant Relay as Bob's Mailbox Relay
  participant Bob
  Alice->>Relay: POST /v1/blobs {tag, expiry, ciphertext}
  Relay->>Relay: validate + store opaque blob
  Note over Bob: offline...
  Bob->>Relay: GET /v1/blobs/{tag}
  Relay-->>Bob: [{ciphertext, expires_at}]
  Bob->>Bob: decrypt

Phase 2: Two-Hop Onion

No single honest relay observes both the upload and the fetch. Alice sends to an entry relay (Carol's node), which decrypts one onion layer and forwards to the mailbox relay (Dave's node) without learning the mailbox tag.

Metadata split: Entry relay sees Alice's upload timing. Mailbox relay sees Bob's fetch timing and the mailbox tag. Neither sees both ends of the same message.

Phase 3: Path Rotation

Each message may use a different entry→mailbox relay pair from the contact's profile descriptors, reducing long-term correlation on a single route.

  P[Contact delivery profile
relay_descriptors] --> R1[Route A: entry₁ → mailbox₁]
  P --> R2[Route B: entry₂ → mailbox₂]
  P --> R3[Route C: entry₃ → mailbox₃]
  MSG1[Message seq=1] --> R1
  MSG2[Message seq=2] --> R2
  MSG3[Message seq=3] --> R3

Phase 5: Direct P2P + Profile Fallback

When both peers are online on the same LAN, the client tries direct HTTP delivery first. If that fails within 2 seconds, it falls back to relay descriptors from the signed profile.

reachable?}
  Direct -->|yes, <2s| P2P[POST to Bob's local port]
  Direct -->|no / timeout| Relay[Use profile relay_descriptors]
  P2P --> Done[Delivered]
  Relay --> Done

Relay authorization

Peers may only advertise a relay in their own signed delivery profile if they are paired with that relay's operator (or operate the relay themselves). Using a contact's advertised relay for delivery does not require pairing with the operator.

Example: Alice is paired with Charlie (relay operator) and advertises Charlie in her profile. Bob pairs with Alice via Charlie rendezvous but does not advertise Charlie — he still sends to Alice via her profile and fetches from Charlie for inbound mail.

Phase 4: Relay Tickets

Public relays can require signed tickets that grant store or forward permission. Tickets are issued during pairing and scoped to a relay name.

sequenceDiagram
  participant Client
  participant Relay
  Client->>Client: build RelayTicket
signed by contact key
  Client->>Relay: POST /v1/blobs + ticket
  Relay->>Relay: verify ticket.permission=store
  alt valid
    Relay-->>Client: 201 stored
  else missing/invalid
    Relay-->>Client: 401 unauthorized
  end
